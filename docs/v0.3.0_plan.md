## 1. 现状与架构可扩展性评估

### 1.1 当前基线（v0.2.0）

- 测试：417 个用例收集；实测 `414 passed / 1 failed / 2 skipped`（失败用例为
  `tests/test_api.py::test_health_endpoint`，期望版本号仍为 `0.1.0`，需先行修复）。
- 覆盖率：整体约 **88%**，核心物理模块（dynamics/models/guidance/phases）多数 ≥90%。
- 代码质量：`flake8`、`mypy` 当前全绿（基于 `.flake8` / `pyproject.toml` 配置）。
- 已完成功能：6-DOF 刚体动力学、导弹弹道整形、Monte Carlo 散布、真实风场地形、
  Tkinter GUI、FastAPI Web API。

### 1.2 架构优势

| 优势 | 说明 |
|------|------|
| 配置统一 | `SimConfig`（pydantic）作为单一事实源，`apply_overrides` 支持点路径覆盖。 |
| Phase 抽象 | `Phase` 基类 + `Powered/Coasting/Reentry/Terminal` 实现，便于插入新阶段。 |
| 坐标系解耦 | `frames.py` 统一 ENU/ECEF/ECI，`state_switch.py` 处理阶段间投影。 |
| 模型插件化 | 大气/引力/气动/风/地形/推进均通过工厂函数装配。 |
| 入口多元 | CLI、GUI、Web API 共享同一 `simulate()` 内核。 |

### 1.3 可扩展性瓶颈

| 瓶颈 | 影响 | 相关文件 |
|------|------|----------|
| `validate_config` 为骨架 | 跨字段业务校验不足，无法提前拦截非法 T/W、推进剂不足等。 | `ballistic_sim/config.py` |
| `phases/builder.py` 过度简化 | rocket/icbm 实际为单级近似，真实多级分离/再入链需手动构造。 | `ballistic_sim/phases/builder.py` |
| 积分器无缓存/无并行 | 每次 `solve_ivp` 重新解析上下文；Monte Carlo process 后端仅做样本级并行。 | `ballistic_sim/simulator.py`, `monte_carlo.py` |
| 制导律种类有限 | PEG 为骨架，缺少闭环比例导引、AAG、再入制导、能量管理。 | `ballistic_sim/guidance/*.py` |
| 可视化静态为主 | 现有 viz 为 matplotlib 纯函数，缺少实时交互 3D/监控面板。 | `ballistic_sim/viz/*.py` |
| 工程化不足 | 无 CI/CD、配置文件驱动未完全贯通、缺少性能基准。 | `.github/workflows/*`（缺） |

## 3. 各阶段详细计划

### 阶段 1：性能优化与工程基座（推荐先启动）

#### 3.1 目标
- 建立可重复的性能基准（benchmark）与回归检测。
- 减少重复计算：缓存大气/风/气动/地形查询，避免 RHS 中反复构造对象。
- 为 Monte Carlo 提供更高效的 batch/process 调度。
- 修复 v0.2.0 遗留的 `test_health_endpoint` 版本断言失败。

#### 3.2 核心文件

| 文件 | 动作 | 说明 |
|------|------|------|
| `tests/test_api.py` | 修改 | `test_health_endpoint` 断言改为 `__version__` 动态值。 |
| `ballistic_sim/simulator.py` | 修改 | `_resolve_dynamics_context` 加 `functools.lru_cache` 或显式缓存；`simulate()` 暴露 `reuse_context` 开关。 |
| `ballistic_sim/dynamics/common.py` | 新增 | `ModelCache` 封装：大气、风、气动系数表按高度/Ma 缓存插值结果。 |
| `ballistic_sim/dynamics/powered_eci.py` | 修改 | RHS 内使用缓存后的 `DynamicContext`，减少引力/气动重复计算。 |
| `ballistic_sim/dynamics/six_dof.py` | 修改 | 同样接入 `ModelCache`；对 `_CD0/_CNa/...` 插值做内层缓存。 |
| `ballistic_sim/monte_carlo.py` | 修改 | batch 后端支持更粗粒度并行；process 后端增加 `maxtasksperchild` 防内存泄漏。 |
| `scripts/benchmark.py` | 新增 | 单发仿真、100 样本 MC、10 样本 6-DOF 的耗时基准，输出 JSON。 |
| `tests/test_performance.py` | 新增 | 轻量级性能断言（如 100 样本 MC 在本地 CI 上 < 60s）。 |
| `tests/test_cache.py` | 新增 | 验证缓存命中、缓存失效、缓存不污染结果。 |

#### 3.3 验收标准

| 维度 | 标准 |
|------|------|
| 测试数 | 417 → ≥447，全部通过，0 failed，≤2 skipped（GPU/显示环境相关）。 |
| 覆盖率 | 整体 ≥85%；新增缓存/基准模块 ≥80%。 |
| flake8/mypy | 全绿，无新增 warning。 |
| 功能验收 | ① `pytest tests/test_api.py::test_health_endpoint` 通过；② `scripts/benchmark.py` 可运行并产出 JSON；③ 100 样本 MC（process 后端）在本地耗时较 v0.2.0 下降 ≥15%。 |

#### 3.4 依赖与风险

- **依赖**：无，可独立启动。
- **风险**：缓存引入状态污染或线程安全问题。
- **回退方案**：缓存默认关闭（`use_cache=False`），出问题可一键回退到当前实现；所有缓存改动通过单元测试验证等价性。

### 阶段 3：制导律扩展

#### 3.9 目标
- 实现并集成多种制导律：
  - 闭环比例导引（PN，适用于拦截/导弹末端）。
  - 迭代显式制导 AAG（Augmented Explicit Guidance，用于火箭上升）。
  - 再入制导（Reentry Guidance，基于阻力加速度剖面/弹道系数）。
  - 能量管理制导（Energy Management，用于亚轨道/返回式飞行器）。
- 每种制导律提供独立模块 + 单元测试 + 与现有 PEG/开环的对比基线。

#### 3.10 核心文件

| 文件 | 动作 | 说明 |

| `ballistic_sim/guidance/proportional_navigation.py` | 新增 | 真比例导引 / 广义比例导引；输入相对位置/速度，输出加速度指令。 |
| `ballistic_sim/guidance/aag.py` | 新增 | 增广显式制导，目标终端状态 (r_T, v_T, γ_T)。 |
| `ballistic_sim/guidance/reentry_guidance.py` | 新增 | 基于阻力加速度剖面的再入制导，输出倾侧角/攻角指令。 |

| `ballistic_sim/config.py` | 修改 | `GuidanceConfig.guidance_law` 扩展为 `(none \| proportional \| peg \| aag \| reentry \| energy)`。 |
| `ballistic_sim/phases/builder.py` | 修改 | 根据 `guidance_law` 装配对应制导对象到 `Phase.guidance`。 |
| `ballistic_sim/dynamics/powered_eci.py` | 修改 | 消费 AAG/PEG 制导指令，转换为推力方向。 |
| `ballistic_sim/dynamics/six_dof.py` | 修改 | 消费再入/能量管理制导的姿态/力矩指令。 |

#### 3.12 依赖与风险

- **依赖**：阶段 2（配置驱动使制导律参数可通过 YAML 注入，builder 自动装配）。
- **风险**：制导律数值收敛性差，导致积分失败或耗时剧增。
- **回退方案**：每种制导律在 `replan`/主循环中设置 `failed` 标志，失败时自动回退到开环 `open_loop` 或 PEG；保留 v0.2.0 开环路径不变。

## 4. 阶段间依赖关系

```text
阶段 1 ──► 阶段 2 ──► 阶段 3 ──┐
       （性能基座）  （配置驱动）  （制导律）    │

                            （预设链完整化） （实时3D可视化） （CI/CD与发布）
```

- **阶段 1 无前置依赖**，可立即启动。
- **阶段 2 依赖阶段 1**：性能缓存使配置校验与 builder 重构后的批量测试可接受。
- **阶段 3 依赖阶段 2**：新制导律依赖配置驱动的 `guidance_law` 字段与 builder 自动装配。
- **阶段 4 依赖阶段 2 和阶段 3**：builder 需支持多级，再入制导可选集成。
- **阶段 5 依赖阶段 1 和阶段 4**：3D 实时播放依赖性能基准，完整轨迹数据依赖阶段 4。
- **阶段 6 依赖阶段 1–5**：发布前所有功能稳定。

## 5. 全局风险与回退方案

| 风险 | 影响 | 回退方案 |
|------|------|----------|
| 性能优化收益不及预期 | 阶段 1 目标无法达成 | 缓存默认关闭；保留原始路径；将性能目标从 "15%" 调整为 "不劣化"。 |
| 新制导律数值不稳定 | 阶段 3/4 集成失败 | 制导律内部设置 `failed` 标志，自动回退到开环/PEG；单元测试单独验证纯算法收敛。 |
| 3D 可视化依赖过重 | 安装/CI 变复杂 | 3D viz 作为可选 extra `[viz3d]`；缺失时降级为 matplotlib 静态图。 |

| 覆盖率因 GUI/3D 下降 | 阶段 5/6 验收卡线 | 合理 skip 显示依赖测试；保持全局 `fail_under=80` 不变。 |
| 真实导弹数据不可得 | 阶段 4 逼真度不足 | 明确使用“教学级公开数据”，并在文档中标注假设与局限性。 |
